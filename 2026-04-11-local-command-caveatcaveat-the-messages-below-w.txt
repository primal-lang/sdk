5. HIGH: Bindings class signature mismatch in runtime code

What is wrong: The design doc at lines 254-261 shows:
@override
Term reduce() {
Map<String, Term> bindingMap = {};
for (final (name, term) in bindings) {
    final Term value = term.substitute(Bindings(bindingMap)).reduce();
    bindingMap[name] = value;
}
return body.substitute(Bindings(bindingMap)).reduce();
}

Why it is wrong: The Bindings constructor takes a Map<String, Term> directly
(const Bindings(this.data)), which matches. However, there's a subtle issue:
the code builds up bindingMap incrementally and wraps it in Bindings each
iteration. Since Bindings stores a reference to the map (not a copy),
mutations to bindingMap after creating Bindings(bindingMap) will affect all
previously created Bindings instances.

The substitute call on iteration 1 creates Bindings(bindingMap) when
bindingMap is empty. On iteration 2, bindingMap has one entry, but the
previous term was substituted with an empty map. This is actually correct for
the sequential scoping semantics (each binding only sees prior bindings), but
the code is sharing the same map instance which could cause issues if Bindings
caches or relies on map immutability.

Looking at lib/compiler/runtime/bindings.dart:5-8:
class Bindings {
final Map<String, Term> data;
const Bindings(this.data);

The const constructor implies the map should be immutable, but the design
passes a mutable map that is modified after construction.

Consequence: The implementation either needs to create a copy of the map each
iteration, or acknowledge this is safe because Bindings only reads from the
map. The current Bindings class does only read, so it's technically safe but
fragile.

- Design doc lines 254-261

- When checking binding x = x + 1, step 1 checks if x (binding name) is in

    - Check the binding's value expression against the current

    - Add the name to `availableParameters`

The shadowing check happens before the value expression check. So the

Why it is wrong: With no separator between bindings, how does the parser know

After parsing x = a, the parser sees y. Is y part of the expression a (making

Cannot parse without lookahead or backtracking that isn't described.

significant"), or
