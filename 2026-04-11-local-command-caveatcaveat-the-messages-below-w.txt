HIGH: usedParameters tracking approach is under-specified

What is wrong: Document lines 343-353 say:
usedParameters Tracking: The existing usedParameters set tracks which function
parameters are referenced. Let bindings are added to availableParameters but 
are NOT function parameters. When an identifier is resolved:
- If it's a function parameter → add to usedParameters
- If it's a let binding → do not add to usedParameters

The document suggests two options but doesn't mandate one. Looking at
semantic_analyzer.dart:247-252:
if (availableParameters.contains(name)) {
usedParameters.add(name);
return SemanticBoundVariableNode(...);
}

Currently, any name in availableParameters gets added to usedParameters. With
let bindings in availableParameters, this would cause false positives
(treating let bindings as "used parameters").

Practical consequence: Unused parameter warnings would break - let binding
usage would suppress warnings for actual unused parameters if they share names
(which is disallowed anyway), but more critically, the logic to distinguish
parameters from let bindings needs explicit implementation.

Missing from spec: The spec doesn't provide the actual code diff for
semantic_analyzer.dart to handle this distinction. It mentions "option 1" but
doesn't commit to it.

MEDIUM: LetTerm.substitute() has subtle correctness gap

What is wrong: Document lines 299-306 show:
@override
Term substitute(Bindings bindings) {
return LetTerm(
    bindings: this.bindings.map((b) => (b.$1,
b.$2.substitute(bindings))).toList(),
    body: body.substitute(bindings),
);
}

The doc claims (lines 321-322):
Since shadowing is disallowed at the semantic level, the substitute() method 
does not need capture-avoiding logic

This is almost correct but has a subtle issue. Consider:
f(x) = let y = x in (let z = 1 in y)

When f is called with argument a, we get:
1. Substitute {x: a} into body
2. Outer let substitutes into x → a, into inner let body → y

This works because y is a let binding name, not a parameter, and bindings are
resolved at reduce() time, not substitute() time. But the implementation in
reduce() (lines 309-319) correctly creates bindingMap incrementally.

However, the document's substitute() propagates substitutions through body
immediately. Consider:
foo(x) = let x = 1 in x  // disallowed - shadowing

This is caught by semantic analysis. But what about:
foo(y) = let x = 1 in y

When called with argument z:
- substitute({y: z}) into LetTerm
- body y → z (BoundVariableTerm.substitute)
- Result: let x = 1 in z

Then reduce():
- bindingMap = {x: 1}
- body.substitute({x: 1}) on z → z (z is not in map)
- Returns z

This is correct! But wait - z is a BoundVariableTerm, and
BoundVariableTerm.substitute() throws if the name isn't found. Looking at
bindings.dart:10-15:
Term get(String name) {
if (data.containsKey(name)) {
    return data[name]!;
} else {
    throw NotFoundInScopeError(name);
}
}

And term.dart:253:
Term substitute(Bindings bindings) => bindings.get(name);

So BoundVariableTerm("z").substitute(Bindings({x: 1})) throws!

This is a bug in the LetTerm.reduce() implementation in the doc. After
evaluating let bindings, the body is substituted with only the let bindings,
not the original function bindings.

Correct implementation should be:
Term reduce() {
Map<String, Term> bindingMap = {};
for (final (name, term) in bindings) {
    final Term substituted = term.substitute(Bindings(Map.of(bindingMap)));
    final Term value = substituted.reduce();
    bindingMap[name] = value;
}
// Body still has BoundVariableTerms from outer scope - they were already
// substituted by the time we got here (during the outer function's apply())
return body.substitute(Bindings(bindingMap)).reduce();
}

Actually wait - looking again at the call flow:
1. CustomFunctionTerm.apply() does substitute(bindings).reduce()
(term.dart:414)
2. This calls LetTerm.substitute(bindings) first, which substitutes all
function parameters into the body
3. Then reduce() is called on the already-substituted term

So by the time LetTerm.reduce() runs, all BoundVariableTerms for function
parameters have already been replaced with values. The remaining
BoundVariableTerms would be for let bindings that haven't been substituted
yet.

Actually no - let bindings create new terms during lowering, not
BoundVariableTerms. Let me re-read...

The document says at line 34:
SemanticBoundVariableNode() => BoundVariableTerm(semanticNode.name),

So let binding usages in the body become BoundVariableTerm too. When we reach
LetTerm.reduce():
- Function parameter BoundVariableTerms have been substituted (from outer
apply())
- Let binding BoundVariableTerms have NOT been substituted yet

So body.substitute(Bindings(bindingMap)) needs to handle let binding names.

But wait - the substitute() in the doc substitutes through the body:
return LetTerm(
bindings: this.bindings.map((b) => (b.$1,
b.$2.substitute(bindings))).toList(),
body: body.substitute(bindings),  // <-- this substitutes function params
);

So after substitute(), the body has function params resolved. Then reduce()
does:
return body.substitute(Bindings(bindingMap)).reduce();

Where bindingMap has let binding names → values. This should work IF and ONLY
IF BoundVariableTerm.substitute doesn't throw for names not in bindings.

But looking at term.dart:253: it DOES throw via bindings.get(name).

This is the bug. Either:
1. Bindings.get() needs to return the original term if name not found
(identity substitution)
2. Or LetTerm.reduce() needs different logic

Actually, checking term.dart line 13:
Term substitute(Bindings bindings) => this;

The base Term.substitute() returns this. Only BoundVariableTerm overrides to
call bindings.get().

The fix would be to make BoundVariableTerm.substitute() return this if not
found:
Term substitute(Bindings bindings) {
if (bindings.data.containsKey(name)) {
    return bindings.get(name);
}
return this;
}

But that changes existing behavior - currently it throws, which is correct for
the existing system where all BoundVariableTerms should be substituted.

The design doc doesn't address this architectural incompatibility.
