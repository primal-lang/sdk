1. `let` is treated as an already-landed prerequisite, but the current branch still has no `let` syntax/IR/runtime support.

Wrong: the doc assumes `let` exists and threads it through grammar, examples, semantic flags, lowering, and tests.  
Why: current expressions are still only `if`/operators/calls/literals; there is no `let` token, AST node, semantic node, or runtime term in this branch.  
Refs: `docs/roadmap/lambda.md:5-11,38,81,221-248,788-816,912-916,1043,1060,1076,1123,1149`; `lib/compiler/syntactic/expression_parser.dart:75-96,286-306`; `lib/compiler/semantic/semantic_node.dart:52-132`; `lib/compiler/lowering/lowerer.dart:24-35`; `docs/compiler/syntactic.md:20-35`; `docs/primal.md:85-90`.  
Consequence: the design is not implementable against current HEAD as written, and multiple examples/tests are for machinery that does not exist yet.  
Missing: either split this into an explicit “land let first” dependency plan, or rewrite the lambda design so it stands on the current branch.

2. The closure semantics overstate “capture by value at creation time” and ignore the current late-bound function-reference architecture.

Wrong: the doc describes captures as creation-time substitution, full stop.  
Why: outer bound variables can be substituted eagerly, but named functions are not captured by value; they lower to `FunctionReferenceTerm` and are resolved through the shared `functions` map at reduce-time. REPL rename/delete mutates that map later.  
Refs: `docs/roadmap/lambda.md:316-336,693-705`; `lib/compiler/lowering/lowerer.dart:30-33`; `lib/compiler/runtime/term.dart:218-231`; `lib/compiler/lowering/runtime_facade.dart:126-139,149-186`.  
Consequence: lambdas that reference named functions will observe later rename/delete/redefinition instead of behaving like fully closed values.  
Missing: specify whether function identifiers inside lambdas are intentionally late-bound; add tests for “create lambda, then rename/delete/redefine referenced function”.

3. Several parser/error expectations are incompatible with the parser shape the document itself proposes.

Wrong: the doc expects `ExpectedTokenError('->')` for `(x) x`, and `InvalidTokenError` for `x -> x + 1` / `1 + (x) -> x`.  
Why: `_checkLambdaStart()` returns `-1` when `)` is not followed by `->`, so those cases fall back to grouped-expression parsing; `Compiler.expression()` then throws `UnexpectedTokenError` on leftover tokens. `x -> x + 1` also parses the `x` first and fails on the trailing arrow.  
Refs: `docs/roadmap/lambda.md:160-166,176-180,453,550-555,581-584,1047-1048`; `lib/compiler/syntactic/expression_parser.dart:295-298`; `lib/compiler/compiler.dart:43-47`; `lib/compiler/errors/syntactic_error.dart:24-26`; `test/compiler/expression_parser_test.dart:1031-1109`.  
Consequence: the specified tests/messages will fail, and implementers could add the wrong parser special cases just to satisfy the doc.  
Missing: tie diagnostics to the actual entrypoint (`Compiler.expression` vs raw `ExpressionParser`) and update expected errors for these invalid forms.

4. The lowering snippet does not compile as written.

Wrong: `_lowerLambda()` names lambdas with `semanticNode.location.line`.  
Why: `Location` exposes `row` and `column`, not `line`.  
Refs: `docs/roadmap/lambda.md:926-929`; `lib/compiler/models/location.dart:1-19`.  
Consequence: the implementation note is mechanically wrong and will fail immediately.  
Missing: choose the actual synthetic-name format and add a test for the lambda name shown in runtime errors / printing.

5. `1 + ((x) -> x)` is presented as “valid” without acknowledging that it is only parse-valid, not runtime-valid.

Wrong: the doc treats the parenthesized lambda as a valid `+` operand.  
Why: `+` accepts `Addable` values only, and `FunctionType` is not an `AddableType`; `OperatorAdd` throws `InvalidArgumentTypesError`.  
Refs: `docs/roadmap/lambda.md:86,1048-1049`; `lib/compiler/library/operators/operator_add.dart:8-16,33-77`; `lib/compiler/models/type.dart:232-245`.  
Consequence: readers and test writers will expect a valid program where evaluation actually fails.  
Missing: label that case as parse-only, or specify the runtime error and add an integration test for `1 + ((x) -> x)`.

6. The diagnostics section overpromises semantic-error location data that the current compiler does not produce.

Wrong: the doc says actual CLI/REPL messages include location information for these semantic errors.  
Why: current semantic errors are plain message objects with no source-location payload; existing formatting tests assert only `Error: ...` text.  
Refs: `docs/roadmap/lambda.md:443-449,547`; `lib/compiler/errors/generic_error.dart:1-13`; `lib/compiler/errors/semantic_error.dart:17-58`; `test/errors/error_formatting_test.dart:418-440,562-590`.  
Consequence: either the doc is inaccurate, or this feature silently expands the semantic-error infrastructure far beyond what is described.  
Missing: decide whether lambda semantic errors remain location-less like current semantic errors, or whether this feature also introduces source-located semantic diagnostics.

7. Lambda printing is specified in a new user-visible format, but the design does not reconcile that with existing function-value rendering.

Wrong: the doc hardcodes lambda `native()` / `toString()` as `"<lambda(x)>"`.  
Why: current function values print via `FunctionTerm.toString()` / `native()` as full signatures, and tests depend on that for named/core functions and lists of functions.  
Refs: `docs/roadmap/lambda.md:649-651,1096-1097`; `lib/compiler/runtime/term.dart:375-381`; `test/runtime/functions/higher_order_test.dart:38-56`.  
Consequence: REPL and collection output becomes inconsistent between named functions and lambdas, and output tests will need deliberate new expectations.  
Missing: specify whether the new format is intentional; add tests for printing a lambda alone, in a list, and mixed with named functions.

8. The `LambdaTerm.substitute()` test table contains a contradictory example.

Wrong: `(x) -> y` with `{y: 5}` is said to produce body `x * 5`.  
Why: neither the setup nor the current substitution model introduces multiplication; substitution only replaces matching names.  
Refs: `docs/roadmap/lambda.md:1101-1105`; `lib/compiler/runtime/term.dart:247-254`.  
Consequence: the test cannot be implemented unambiguously.  
Missing: change the setup to `(x) -> x * y`, or change the expected result to `5`.

**Open Questions**
- The doc assumes lambda-parameter shadowing should be banned (`docs/roadmap/lambda.md:283-299`). Current resolution logic simply prefers available parameters over function names (`lib/compiler/semantic/semantic_analyzer.dart:247-258,346-367`), and existing tests already allow parameter-over-function shadowing (`test/compiler/semantic_analyzer_test.dart:237-244`). If the no-shadowing rule is intentional, it should be stated as a language rule, not just an implementation convenience.
- The doc assumes `let`-style scope flags like `isLetBinding` and `letBindingNames` already exist. That assumption does not match current HEAD, so the design needs either a staged dependency plan or a different base revision.
